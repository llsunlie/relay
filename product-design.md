# Relay：Agent 产品设计文档

## 一、产品定位

### 一句话描述

**每个人的数字收发秘书**——一个运行在后台的智能代理，接管所有"做完之后该交给谁"的认知负担，让人只关心"做"。

### 不是什么东西

- **不是一个独立 App**——用户不需要打开一个新 App 来管理任务
- **不是一个消息客户端**——它不创造新的沟通渠道，而是利用现有渠道
- **不是一个工作流引擎**——它不定义任务怎么做，只负责做完之后的交付

### 是什么

一个**智能路由和转换层**，位于用户日常工具之上、通信基础设施之上：

```
┌──────────────────────────────────────────────────────┐
│                    用户日常工具层                      │
│   VS Code  │  飞书  │  WPS  │  邮件  │  微信  │ ...  │
└──────────────┬────────────────────────────────────────┘
               │  产出事件（文件保存、分享意图、完成信号）
┌──────────────▼────────────────────────────────────────┐
│                   Agent 智能代理层                     │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────────┐ │
│  │  Sender   │  │  Inbox   │  │  规则引擎 & 关系图谱  │ │
│  └────┬─────┘  └────┬─────┘  └──────────────────────┘ │
│       │              │                                  │
│  ┌────▼──────────────▼─────────────────────────────┐  │
│  │              协商协议 & 传输总线                   │  │
│  └─────────────────────────────────────────────────┘  │
└──────────────┬────────────────────────────────────────┘
               │  投递（适配器模式）
┌──────────────▼────────────────────────────────────────┐
│                 通信基础设施层                          │
│   邮件  │  IM  │  Webhook  │  API  │  文件系统  │ ...  │
└──────────────────────────────────────────────────────┘
```

## 二、目标用户画像

| 画像 | 场景特征 | 核心痛点 |
|------|---------|---------|
| **导师/老师** | 同时带多个学生，收作业、论文、实验报告，周期性的材料收集 | 多个学生发到不同渠道（微信、邮件、课程平台），分散混乱 |
| **管理者** | 收下属周报、审批报销请假、汇总汇报材料 | 收件渠道多，催收成本高，格式不统一 |
| **学生** | 向多位老师交不同类型作业，填各种申请 | 搞不清每个任务该交谁、走哪个系统、截止时间是什么 |
| **知识工作者** | 跨部门协作，产出需要流转多方 | 交付链路长，不知道谁在看、流转到哪了 |
| **自由职业者** | 多客户，不同客户不同合同/发票/素材要求 | 交付物形态多样，不同客户格式偏好不一 |

## 三、核心概念定义

### 3.1 交付物（Delivery）

用户在任意工具中完成的一项数字产出。包括但不限于：

- 文件（文档、表格、图片、代码仓库）
- 消息（一段文本、一个链接）
- 链接/指针（活的引用，指向持续更新的源）
- 操作指令（一个需要对方处理的动作，如审批、批改）
- 状态信号（"我的部分已就绪"）

### 3.2 交付意图（Delivery Intent）

用户表达"这个需要交出去"的最小动作：

| 意图层级 | 用户动作 | 自动程度 |
|---------|---------|---------|
| L0 手动 | 打开交付面板，指定收件人，点发送 | 0% |
| L1 半自动 | 分享菜单选"交付"，选模板（"这是周报"） | 30% |
| L2 规则自动 | 放到约定目录/标签，自动匹配规则 | 80% |
| L3 全自动 | Agent 检测到完成信号（文档保存、commit push、表单提交），自动交付 | 100% |

目标是让大多数日常交付从 L1/L2 起步，逐步过渡到 L3。

### 3.3 交付规则（Delivery Rule）

一条交付规则回答三个问题：

```
当 [产出类型] 出现在 [来源] 时，交付给 [收件角色]，
通过 [通道]，以 [格式偏好]，在 [时机约束] 下完成。
```

示例：

```yaml
rule:
  name: "周报交付"
  trigger:
    type: "文件创建"
    pattern: "~/relay-outbox/weekly/*.md"
    metadata:
      content_type: "周报"
      period: "weekly"
  target:
    role: "直属上级"
    resolve_by: "关系图谱"
  delivery:
    channel: ["对方InboxAPI", "企业邮箱(备用)"]
    format: "对方偏好"
    timing:
      strategy: "周期窗口"
      window: "每周五 08:00-20:00"
      batch: true  # 同一周期的同类交付物批量发送
    lifecycle:
      retract_window: "接收方未读前可撤回"
      expiry: null
  on_completion:
    - action: "通知交付方"
      message: "周报已送达"
```

### 4.2 本地 Agent 模块详解

#### 4.2.1 产出监听器（Watcher）

```
监听源                     检测机制
─────────────────────────────────────────
文件系统目录               inotify / FSEvents
指定 App 的完成事件         App 插件 / Shortcuts / URL Scheme
剪贴板                     Clipboard API (需要用户授权)
系统分享菜单               Share Extension
浏览器下载完成             Browser Extension
Git 仓库 push              Git Hook / GitHub Webhook
```

Watcher 不传输内容，只捕获**产出事件**（元信息：路径、类型、时间、上下文）。实际内容在交付确认后再读取传输。

#### 4.2.2 交付打包器（Packager）

接收到产出事件后，Packager 负责：

1. **内容识别**：AI 分析文件内容，确定产出类型（周报 vs 设计稿 vs 发票）
2. **元数据提取**：从文件名、内容、上下文提取关键信息（日期、项目、版本）
3. **转换适配**：根据目标Inbox声明偏好进行格式转换
4. **构建交付包**：组装标准化的交付数据包

```typescript
interface DeliveryPackage {
  id: string;
  source: {
    userId: string;
    agentId: string;
  };
  content: {
    type: ContentType;          // "weekly_report" | "design_draft" | "invoice" ...
    payload: Payload;           // 实际内容或指针
    metadata: Record<string, any>;
  };
  presentation: {
    formats: string[];          // 提供的格式列表
    defaultView: string;        // 默认展示格式
  };
  lifecycle: {
    createdAt: timestamp;
    expiresAt?: timestamp;
    retractableUntil?: timestamp;
    version: string;            // 语义版本（活的交付物用）
  };
  routing: {
    targetRole: string;
    targetUserId?: string;      // 解析前为空
    channel?: string;
    priority: "low" | "normal" | "high" | "urgent";
  };
  actions: Action[];            // 接收方可以执行的操作
}
```

#### 4.2.3 本地收件箱（Inbox）

Inbox在本地侧的呈现。设计原则：

- **不是另一个"待办 App"**——收件箱是聚合视图，不是新的工作入口
- **适配用户节奏**——支持"早晨摘要模式""实时模式""免打扰模式"
- **轻度交互**——大部分东西看一眼即可，少部分需要操作的提供快捷动作

收件箱的呈现形态：

```
┌─────────────────────────────────────────┐
│  📥 今日收件                    [设置]   │
├─────────────────────────────────────────┤
│                                         │
│  🔴 需要处理（2）                        │
│  ├── 张三的请假申请                      │
│  │   需要你审批 │ 5分钟前                 │
│  │   [同意] [拒绝] [稍后]               │
│  │                                      │
│  ├── 李四的实验报告 v3                   │
│  │   等待批改 │ 2小时前                   │
│  │   [打开] [标记已阅]                  │
│                                         │
│  🟡 待查看（5）                          │
│  ├── 本周周报汇总                        │
│  │   5位下属已提交 │ 今天 09:30           │
│  │   [查看汇总] [逐份查看]              │
│  │                                      │
│  ├── 预算审批通过通知                    │
│  │   来自：财务系统 │ 昨天               │
│  │   ...                                │
│                                         │
│  ⚪ 已自动归档（12）                      │
│     系统通知、收据、回执等               │
│                                         │
└─────────────────────────────────────────┘
```

## 五、核心功能设计

### 5.1 关系图谱

关系图谱是Sender解析"交给谁"的基础数据。它描述人与人之间的**交付关系**（不是社交关系）。

#### 关系类型

```yaml
relationship:
  types:
    - supervisor:      # 汇报关系
        direction: upward
        delivery_types: [weekly_report, leave_request, reimbursement, review_request]
    - supervisee:      # 被汇报关系（反向自动推导）
        direction: downward
        delivery_types: []  # 通常是接收方
    - teacher:         # 指导关系
        direction: upward
        delivery_types: [homework, thesis_draft, experiment_report, consultation]
    - student:         # 被指导关系
        direction: downward
    - client:          # 客户关系
        direction: outward
        delivery_types: [deliverable, invoice, contract, progress_report]
    - reviewer:        # 评审关系
        direction: upward
        delivery_types: [code_review, design_review, document_review]
    - peer:            # 协作关系
        direction: mutual
        delivery_types: [handoff, shared_draft, meeting_notes]
```

#### 关系来源

| 来源 | 方式 |
|------|------|
| 组织架构同步 | 对接企业通讯录/HR 系统 |
| 用户手动建立 | "我和张三是导师-学生关系" |
| AI 推断 | 从邮件往来、日历、协作频率推断并建议，用户确认 |
| 模板 | 预置常见关系模板（师生、上下级、甲乙方） |

### 5.2 规则引擎

规则引擎是Sender自动决策的核心，它决定"这个产出触发哪条交付规则"。

#### 规则匹配流程

```
产出事件 → 类型识别 → 规则匹配 → 收件人解析 → 通道选择 → 协商 → 投递
              │           │            │            │          │
              ▼           ▼            ▼            ▼          ▼
           AI分类器    规则库索引    关系图谱    Inbox注册表  协商协议
```

#### 规则 DSL

```yaml
# 规则示例：学生交实验报告
rule:
  id: "student-lab-report"
  name: "实验报告交付"
  
  # 触发条件：AND 关系
  when:
    content_type: "experiment_report"
    source:
      - folder: "~/courses/*/lab-reports/"
      - app: "jupyter"
      - share_sheet: true
    context:
      course_id: "extracted_from_folder_name"
      
  # 交付动作
  then:
    deliver_to:
      role: "teacher"
      filter: "teaching_course == {{course_id}}"
      resolve_deadline: "from_syllabus"
    
    # 交付策略
    strategy:
      format: "inbox_preference"      # 按接收方偏好转换
      channel: ["direct", "lms_api"]     # 优先直连，备选 LMS API
      timing:
        type: "deadline_aware"           # 感知截止日期
        remind: "24h_before_deadline"    # 临近截止提醒
        auto_submit: true                # 截止前自动提交
        
    # 交付后
    on_delivery:
      - notify_sender: "实验报告已送达{{teacher.name}}"
      - notify_receiver: "收到{{sender.name}}的实验报告"
      - update_course_tracker: true
      
    # 接收方可执行的操作
    inbox_actions:
      - mark: { options: [优秀, 通过, 需修改] }
      - annotate: { format: "inline_comment" }
      - return: { reason: "需要修改" }
      - forward: { to: "助教" }
```

### 5.3 协商协议

这是整个系统最具差异化的部分。Sender和Inbox之间在传输前进行语义层协商。

#### 协商流程

```
交付方 Agent                              接收方 Agent
    │                                         │
    │  ─── CAPABILITY_QUERY ──────────────►  │
    │    {                                     │
    │      "delivery_type": "weekly_report",   │
    │      "available_formats": ["md","pdf"],  │
    │      "size_bytes": 102400,               │
    │      "proposed_timing": "now",           │
    │      "priority": "normal",               │
    │      "retractable": true                 │
    │    }                                     │
    │                                         │
    │  ◄─── CAPABILITY_RESPONSE ────────────  │
    │    {                                     │
    │      "accepted_formats": ["md"],         │
    │      "preferred_channel": "agent_api",   │
    │      "load_status": "low",               │
    │      "accept_timing": "now",             │
    │      "transform_hints": {                │
    │        "template": "weekly_summary",     │
    │        "language": "zh-CN"               │
    │      },                                  │
    │      "receiver_ready": true              │
    │    }                                     │
    │                                         │
    │  ─── DELIVERY_PUSH ──────────────────►  │
    │    {                                     │
    │      "package": <DeliveryPackage>,       │
    │      "negotiation_id": "neg_xxx",        │
    │      "checksum": "sha256..."             │
    │    }                                     │
    │                                         │
    │  ◄─── DELIVERY_ACK ───────────────────  │
    │    {                                     │
    │      "status": "received",               │
    │      "estimated_processing": "2h",       │
    │      "auto_actions_taken": [             │
    │        "classified_to_inbox:需要处理",    │
    │        "ai_summary_generated"            │
    │      ]                                   │
    │    }                                     │
```

#### Inbox 能力声明（Inbox Capability Profile）

```typescript
interface InboxProfile {
  userId: string;
  
  // 格式偏好
  formatPreferences: {
    [contentType: string]: {
      preferred: string[];       // ["markdown", "json"]
      rejected: string[];        // ["docx"] — 不收 Word
      transformTemplate?: string; // 可选的转换模板
    }
  };
  
  // 通道 & 负载
  channels: {
    agent_api: { endpoint: string; status: "active" | "busy" | "offline" };
    email?: { address: string; quotaRemaining: number };
    im_bot?: { platform: string; webhook: string };
    webhook?: { url: string; secret: string };
  };
  
  // 接收节律
  deliveryWindows: {
    [contentType: string]: {
      strategy: "anytime" | "work_hours" | "scheduled";
      windows?: TimeWindow[];
      batching: "none" | "hourly" | "daily" | "weekly";
    }
  };
  
  // AI 预处理偏好
  preprocessing: {
    autoSummarize: boolean;       // 自动摘要
    extractActionItems: boolean;  // 提取待办项
    anomalyDetection: boolean;    // 异常检测
    languagePreference: string;   // 摘要语言
  };
  
  // 接收方可提供的操作
  supportedActions: ActionType[];
}
```

### 5.4 交付生命周期

```
[创建] ──→ [协商中] ──→ [已投递] ──→ [已接收] ──→ [已处理] ──→ [归档]
              │             │          │            │            │
              │             ▼          ▼            ▼            ▼
              │          [协商失败]  [投递失败]  [拒收]      [过期]
              │             │          │            │            │
              └──── 重试/降级 ──────────┘            │            │
                              │                      │            │
                              └── 通知发送方 ◄────────┘            │
                                     │                            │
                                     └── 重新交付 ────────────────┘
```

每个状态变更都记录在云端状态追踪服务中，发送方可查看交付进度。

## 六、用户交互设计

### 6.1 交付侧交互

#### 入口点矩阵

| 场景 | 入口 | 交互 |
|------|------|------|
| 文件已完成 | 系统分享菜单 → "交付" | 选模板 → 确认 |
| 代码已推送 | Git 提交信息包含 `[deliver]` 标记 | 自动触发 |
| 拖拽到托盘 | 拖文件到系统托盘图标 | 弹出交付确认弹窗 |
| 文件夹自动 | 保存到约定目录 | 无交互，静默交付 |
| 语音 | "帮我交下周报" | AI 对话式交付 |

#### 交付确认弹窗（需要人工确认的场景）

```
┌─────────────────────────────────────────┐
│  📤 交付本周周报                         │
│                                         │
│  文件: 周报-2026W20.md                  │
│  识别: 周报（5月第3周）                 │
│                                         │
│  交付给: 李总（直属上级）               │
│  格式: Markdown → 转为对方偏好          │
│  时机: 现在发送                         │
│                                         │
│  ┌─────────────────────────────┐        │
│  │ 📝 留言（可选）              │        │
│  │ 本周重点：完成了A项目的设...  │        │
│  └─────────────────────────────┘        │
│                                         │
│     [取消]          [确认交付]           │
└─────────────────────────────────────────┘
```

### 6.2 接收侧交互

Inbox以**每日摘要通知**为主要触达方式（避免实时骚扰）：

```
┌─────────────────────────────────────────┐
│  📥 今日摘要 — 5月19日 周二              │
│                                         │
│  ════════ 需要你处理的 ════════         │
│  ▸ 张三的请假申请                        │
│    离岗 5/20-5/22，需要你审批            │
│    [处理]                               │
│                                         │
│  ▸ 2份待批改的实验报告                   │
│    李四 — 光学实验                       │
│    王五 — 电磁学实验（迟交）             │
│    [逐份查看] [批量处理]               │
│                                         │
```

## 七、技术设计关键决策

### 7.1 内容寻址与引用

**原则**：尽量传引用不传副本。只有对方需要离线查看或格式转换时才传完整内容。

```typescript
type Payload =
  | { type: "file"; path: string; hash: string }
  | { type: "link"; url: string; live: boolean }     // live=true 表示活的交付物
  | { type: "inline"; data: Buffer; mimeType: string }
  | { type: "stream"; endpoint: string; protocol: "grpc" | "rest" }
  | { type: "reference"; pointer: string; resolver: string }
```

### 7.2 格式转换引擎

可插拔的转换管线：

```
原始格式 → [结构提取] → [语义模型] → [目标模板] → 目标格式
                           │
                    中间表示层 (IR)
                    纯语义，无格式
```

| 转换路径 | 方法 |
|---------|------|
| Markdown → PDF | Pandoc |
| Markdown → 邮件正文 | 模板渲染 |
| CSV → 图表 PNG | AI 自动可视化 |
| 文本 → 语音摘要 | TTS 引擎 |
| 长文档 → 短摘要 | LLM 摘要 |
| 任意文本 → 对方语言 | 翻译引擎 |

### 7.3 安全与隐私

| 层面 | 措施 |
|------|------|
| 内容加密 | 传输层 TLS 1.3，落盘 AES-256 |
| 访问控制 | 只有协商确认的接收方能解密交付包 |
| 本地优先 | 内容分析和格式转换尽量在本地完成，不上传云端 |
| 关系图谱 | 端到端加密存储，云端只存加密后的图谱 |
| 时效销毁 | 支持交付物在指定时间后自动从接收方删除 |
| 撤回 | 接收方未读前可撤回，撤回后云端和本地均删除痕迹 |
| 审计日志 | 交付全链路可审计，但日志不入云端，存在本地 |

### 7.4 离线与异步

- 交付方可离线创建交付意图，联网后自动完成协商和投递
- 接收方离线时，交付包缓存在云端交付总线，上线后推送
- 协商信息（Inbox能力声明）缓存本地，定期更新

## 八、分阶段落地计划

### Phase 1：基础交付（MVP，4-6 周）

**目标**：验证"不改变工作习惯的交付"是否可行

| 能力 | 范围 |
|------|------|
| 系统分享菜单集成 | 桌面端：文件右键"交付" |
| 基础关系图谱 | 用户手动建立关系（"A 是 B 的上级"） |
| 基础规则引擎 | 5-10 个预置模板（周报交付、作业提交、审批申请等） |
| 单一通道 | 邮件投递（覆盖最广） |
| 交付状态追踪 | 发送方可见"已发送/已阅读" |

**交付物**：桌面端 Agent（macOS/Windows 托盘应用）

### Phase 2：自动识别 + Inbox（8-12 周）

| 能力 | 范围 |
|------|------|
| 文件夹监听 | 约定目录自动交付 |
| AI 内容识别 | 自动判断产出类型 |
| 本地收件箱 | 聚合视图 + 每日摘要 |
| 格式转换 | Markdown ↔ 邮件、CSV → 图表等基础转换 |
| 多通道 | 邮件 + IM Bot（企业微信/飞书/Slack） |
| 关系导入 | 对接企业通讯录 |

**交付物**：接收端体验上线，形成完整闭环

### Phase 3：协商协议 + 生态（16-24 周）

| 能力 | 范围 |
|------|------|
| 协商协议 | Sender/Inbox间自动协商格式、时机、通道 |
| 活的交付物 | 交付指针，接收方持续订阅更新 |
| 可执行交付物 | 接收方可直接操作（审批、批改、驳回） |
| 反向交付 | 接收方声明需求，交付方按规约提交 |
| SDK/API | 允许第三方 App 接入交付协议 |
| 影子收件人 | AI 自动摘要、异常检测、待办提取 |

**交付物**：开放协议文档 + SDK，生态可扩展

## 九、关键指标

| 指标 | 定义 | 目标值 |
|------|------|--------|
| 交付成功率 | 发起交付后最终送达的比例 | > 99% |
| 用户操作次数 | 完成一次交付需要用户点击的次数 | Phase 2 后降到 0-1 次 |
| 交付时间 | 从产出完成到对方收到的时间 | 实时交付 < 5s |
| 格式转换准确率 | 自动转换后被用户修改的比例 | < 5% 需手动修正 |
| 规则匹配准确率 | 自动匹配的收件人正确的比例 | > 95% |
| 收件箱处理率 | 接收方对推送内容的操作率（非忽略率） | > 60% |

## 十、开放问题

1. **身份统一**：用户如何跨组织边界使用？用手机号？邮箱？还是需要一个开放的身份协议？（类似 Email 的 `user@domain`）
2. **关系图谱的可移植性**：换公司/换学校后，关系图谱能否迁移？怎样设计图谱的生命周期？
3. **垃圾交付问题**：如果人人都能无成本交付，如何防止"交付泛滥"？接收方需要什么样的过滤/限流机制？
4. **遗留系统对接**：大量学校/企业有内部系统（课程平台、OA），如何在不改造它们的前提下与 Agent 对接？
5. **冷启动**：新用户没有任何规则和关系图谱时，体验如何不打断？
6. **商业模型**：谁为这个基础设施付费？个人用户免费 + 企业版收费？还是按交付量计费？
