# Relay：Agent 产品设计文档

## 一、产品定位

### 一句话描述

**每个人的数字收发秘书**——一个运行在后台的智能代理，接管所有"做完之后该交给谁"的认知负担，让人只关心"做"。

### 不是什么东西

- **不是一个独立 App**——用户不需要打开一个新 App 来管理任务
- **不是一个消息客户端**——它不创造新的沟通渠道，而是利用现有渠道
- **不是一个工作流引擎**——它不定义任务怎么做，只负责做完之后的交付

### 是什么

一个**智能路由和转换层**，位于用户日常工具之上、通信基础设施之上：

```
┌──────────────────────────────────────────────────────┐
│                    用户日常工具层                      │
│   VS Code  │  飞书  │  WPS  │  邮件  │  微信  │ ...  │
└──────────────┬────────────────────────────────────────┘
               │  产出事件（文件保存、分享意图、完成信号）
┌──────────────▼────────────────────────────────────────┐
│                   Agent 智能代理层                     │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────────┐ │
│  │  Sender   │  │  Inbox   │  │  规则引擎 & 交付对象  │ │
│  └────┬─────┘  └────┬─────┘  └──────────────────────┘ │
│       │              │                                  │
│  ┌────▼──────────────▼─────────────────────────────┐  │
│  │              协商协议 & 传输总线                   │  │
│  └─────────────────────────────────────────────────┘  │
└──────────────┬────────────────────────────────────────┘
               │  投递（适配器模式）
┌──────────────▼────────────────────────────────────────┐
│                 通信基础设施层                          │
│   邮件  │  IM  │  Webhook  │  API  │  文件系统  │ ...  │
└──────────────────────────────────────────────────────┘
```

## 二、目标用户画像

| 画像 | 场景特征 | 核心痛点 |
|------|---------|---------|
| **导师/老师** | 同时带多个学生，收作业、论文、实验报告，周期性的材料收集 | 多个学生发到不同渠道（微信、邮件、课程平台），分散混乱 |
| **管理者** | 收下属周报、审批报销请假、汇总汇报材料 | 收件渠道多，催收成本高，格式不统一 |
| **学生** | 向多位老师交不同类型作业，填各种申请 | 搞不清每个任务该交谁、走哪个系统、截止时间是什么 |
| **知识工作者** | 跨部门协作，产出需要流转多方 | 交付链路长，不知道谁在看、流转到哪了 |
| **自由职业者** | 多客户，不同客户不同合同/发票/素材要求 | 交付物形态多样，不同客户格式偏好不一 |

## 三、核心概念定义

### 3.1 交付物（Delivery）

用户在任意工具中完成的一项数字产出。包括但不限于：

- 文件（文档、表格、图片、代码仓库）
- 消息（一段文本、一个链接）
- 链接/指针（活的引用，指向持续更新的源）
- 操作指令（一个需要对方处理的动作，如审批、批改）
- 状态信号（"我的部分已就绪"）

### 3.2 交付意图（Delivery Intent）

用户表达"这个需要交出去"的最小动作：

| 意图层级 | 用户动作 | 自动程度 |
|---------|---------|---------|
| L0 手动 | 打开交付面板，指定收件人，点发送 | 0% |
| L1 半自动 | 分享菜单选"交付"，选模板（"这是周报"） | 30% |
| L2 规则自动 | 放到约定目录/标签，自动匹配规则 | 80% |
| L3 全自动 | Agent 检测到完成信号（文档保存、commit push、表单提交），自动交付 | 100% |

目标是让大多数日常交付从 L1/L2 起步，逐步过渡到 L3。

### 3.3 交付规则（Delivery Rule）

一条交付规则回答三个问题：

```
当 [产出类型] 出现在 [来源] 时，交付给 [收件角色]，
通过 [通道]，以 [格式偏好]，在 [时机约束] 下完成。
```

示例：

```yaml
rule:
  name: "周报交付"
  trigger:
    type: "文件创建"
    pattern: "~/relay-outbox/weekly/*.md"
    metadata:
      content_type: "周报"
      period: "weekly"
  target:
    role: "直属上级"
    resolve_by: "关系图谱"
  delivery:
    channel: ["对方InboxAPI", "企业邮箱(备用)"]
    format: "对方偏好"
    timing:
      strategy: "周期窗口"
      window: "每周五 08:00-20:00"
      batch: true  # 同一周期的同类交付物批量发送
    lifecycle:
      retract_window: "接收方未读前可撤回"
      expiry: null
  on_completion:
    - action: "通知交付方"
      message: "周报已送达"
```

### 4.2 本地 Agent 模块详解

#### 4.2.1 产出监听器（Watcher）

```
监听源                     检测机制
─────────────────────────────────────────
文件系统目录               inotify / FSEvents
指定 App 的完成事件         App 插件 / Shortcuts / URL Scheme
剪贴板                     Clipboard API (需要用户授权)
系统分享菜单               Share Extension
浏览器下载完成             Browser Extension
Git 仓库 push              Git Hook / GitHub Webhook
```

Watcher 不传输内容，只捕获**产出事件**（元信息：路径、类型、时间、上下文）。实际内容在交付确认后再读取传输。

#### 4.2.2 交付打包器（Packager）

接收到产出事件后，Packager 负责：

1. **内容识别**：AI 分析文件内容，确定产出类型（周报 vs 设计稿 vs 发票）
2. **元数据提取**：从文件名、内容、上下文提取关键信息（日期、项目、版本）
3. **转换适配**：根据目标Inbox声明偏好进行格式转换
4. **构建交付包**：组装标准化的交付数据包

```typescript
interface DeliveryPackage {
  id: string;
  source: {
    userId: string;
    agentId: string;
  };
  content: {
    type: ContentType;          // "weekly_report" | "design_draft" | "invoice" ...
    payload: Payload;           // 实际内容或指针
    metadata: Record<string, any>;
  };
  presentation: {
    formats: string[];          // 提供的格式列表
    defaultView: string;        // 默认展示格式
  };
  lifecycle: {
    createdAt: timestamp;
    expiresAt?: timestamp;
    retractableUntil?: timestamp;
    version: string;            // 语义版本（活的交付物用）
  };
  routing: {
    targetRole: string;
    targetUserId?: string;      // 解析前为空
    channel?: string;
    priority: "low" | "normal" | "high" | "urgent";
  };
  actions: Action[];            // 接收方可以执行的操作
}
```

#### 4.2.3 本地收件箱（Inbox）

Inbox在本地侧的呈现。设计原则：

- **不是另一个"待办 App"**——收件箱是聚合视图，不是新的工作入口
- **适配用户节奏**——支持每日摘要模式、免打扰模式
- **轻度交互**——大部分东西看一眼即可

收件箱的呈现形态：

```
┌─────────────────────────────────────────┐
│  📥 收件箱                    [搜索]     │
├─────────────────────────────────────────┤
│                                         │
│  🔴 待处理                               │
│  ├── 📄 周报-2026W20.md                 │
│  │   李总 · 今天 09:30                   │
│  │   AI摘要：本周完成A项目设计...         │
│  │                                      │
│  ├── 📝 请假申请                         │
│  │   张三 · 昨天 17:00                   │
│  │   摘要：5/20-5/22离岗                 │
│  │                                      │
│  ⚪ 已归档                               │
│  ├── 📊 实验报告-光学.docx               │
│  │   李四 · 3天前                        │
│  │   ...                                │
│                                         │
└─────────────────────────────────────────┘
```

## 五、核心功能设计

### 5.1 交付对象

交付对象是 Sender 解析"交给谁"的基础数据。交付对象可以是**个人**或**系统**。

#### 个人

| 字段 | 说明 |
|------|------|
| 姓名 | 必填 |
| 邮箱 | 必填，校验格式 |
| 关系类型 | 上级 / 下属 / 老师 / 学生 / 客户 / 协作伙伴 |

#### 系统

| 字段 | 说明 |
|------|------|
| 系统名称 | 必填 |
| 通道地址 | 必填，如 Webhook URL |

#### 数据来源

| 来源 | 方式 |
|------|------|
| 用户手动建立 | 填写姓名、邮箱、关系类型或通道地址 |
| 企业通讯录导入 | LDAP / Microsoft 365 / 飞书 / 企业微信自动导入 |
| 组织树推导 | 直接上级 →「上级」、直接下级 →「下属」、同部门同级 →「协作伙伴」 |

### 5.2 规则模板

规则模板是 Sender 自动预填交付信息的基础。根据文件名关键词匹配交付类型和收件人角色。

#### 匹配流程

```
文件名 → 关键词匹配 → 预填交付类型 + 收件人角色 → 从交付对象列表解析具体人 → 交付窗口展示
```

#### 预置规则

| 文件名关键词 | 预填交付类型 | 预填收件人角色 |
|-------------|------------|--------------|
| `周报` 开头 | 周报 | 上级 |
| `作业` 开头 | 作业提交 | 老师 |
| `请假` 开头 | 请假申请 | 上级 |
| `报销` 开头 | 报销申请 | 上级 |
| 无匹配 | 通用 | 不预填 |

命中后从交付对象中查找对应角色的个人，多个同角色时预填第一个。用户仍可在交付窗口中手动修改预填结果。

### 5.3 协商协议

Sender 和 Inbox 之间在传输前进行语义层协商。

#### 协商流程

```
交付方 Agent                              接收方 Agent
    │                                         │
    │  ─── 能力查询 ──────────────────────►  │
    │    { 交付类型, 可用格式, 文件大小 }        │
    │                                         │
    │  ◄─── 能力响应 ───────────────────────  │
    │    { 接受格式, 偏好通道, 接收窗口 }        │
    │                                         │
    │  ─── 交付推送 ──────────────────────►  │
    │    { 交付包, 协商ID }                     │
    │                                         │
    │  ◄─── 交付确认 ──────────────────────   │
    │    { 状态: 已接收 }                       │
```

#### Inbox 能力声明

```typescript
interface InboxProfile {
  userId: string;
  
  // 接受的交付类型
  acceptedTypes: ContentType[];
  
  // 格式偏好
  formatPreferences: {
    [contentType: string]: string[];  // ["markdown", "pdf"]
  };
  
  // 可用通道
  channels: {
    email?: { address: string };
    im_bot?: { platform: string };
    webhook?: { url: string };
  };
  
  // 接收窗口
  deliveryWindows: {
    strategy: "anytime" | "work_hours";
  };
}
```

### 5.4 交付生命周期

```
[创建] ──→ [协商中] ──→ [已投递] ──→ [已接收] ──→ [归档]
              │             │            
              │             ▼            
              │          [投递失败]      
              │             │            
              └──── 重试/降级 ─┘
```

每个状态变更记录在本地交付历史中，发送方可在交付记录中查看进度。

## 六、用户交互设计

### 6.1 交付侧交互

#### 入口点矩阵

| 场景 | 入口 | 交互 |
|------|------|------|
| 文件已完成 | 系统分享菜单 → "交付" | 选模板 → 确认 |
| 代码已推送 | Git 提交信息包含 `[deliver]` 标记 | 自动触发 |
| 拖拽到托盘 | 拖文件到系统托盘图标 | 弹出交付确认弹窗 |
| 文件夹自动 | 保存到约定目录 | 无交互，静默交付 |
| 语音 | "帮我交下周报" | AI 对话式交付 |

#### 交付窗口（需要人工确认的场景）

```
┌─────────────────────────────────────────┐
│  确认交付                               │
│                                         │
│  文件: 周报-2026W20.md (12KB)           │
│                                         │
│  交付类型: [周报               ▼]       │
│  收件人:   [李总（上级）        ▼]       │
│                                         │
│  摘要: 本周完成A项目设计、B模块开发...   │
│  ┌─────────────────────────────┐        │
│  │ 补充说明（可选）              │        │
│  └─────────────────────────────┘        │
│                                         │
│     [取消]          [确认交付]           │
└─────────────────────────────────────────┘
```

### 6.3 Agent 常驻形态

| 平台 | 常驻形态 | 说明 |
|------|---------|------|
| 桌面端 | 系统托盘图标 + 少量弹窗 | 右键菜单快速交付、状态指示灯 |
| 移动端 | 通知栏 Widget + 分享菜单 | 分享菜单里的"交付"选项 |
| Web | 浏览器扩展 | 捕捉下载、Web App 产出 |

Agent 不是需要"打开"的东西——它永远在后台，你只在需要确认或查看时才与它交互。

## 七、技术设计关键决策

### 7.1 内容寻址与引用

**原则**：尽量传引用不传副本。只有对方需要离线查看或格式转换时才传完整内容。

```typescript
type Payload =
  | { type: "file"; path: string; hash: string }
  | { type: "link"; url: string; live: boolean }     // live=true 表示活的交付物
  | { type: "inline"; data: Buffer; mimeType: string }
  | { type: "stream"; endpoint: string; protocol: "grpc" | "rest" }
  | { type: "reference"; pointer: string; resolver: string }
```

### 7.2 格式转换引擎

发送方发送前，若接收方有格式偏好且与当前格式不同，自动转换。

**v0.1.1 转换路径**：

| 源格式 | 目标格式 |
|--------|---------|
| Markdown | HTML、PDF、TXT |
| DOCX | Markdown、PDF |
| TXT | Markdown |
| CSV | Markdown 表格 |

转换失败时通知用户，可选择以原格式继续发送或取消。

### 7.3 安全与隐私

| 层面 | 措施 |
|------|------|
| 内容加密 | 传输层 TLS 1.3，落盘 AES-256 |
| 本地优先 | 内容分析和格式转换尽量在本地完成，不上传云端 |
| 交付对象 | 本地存储，不上传云端 |
| 撤回 | Agent 直连通道下，接收方未读前可撤回 |
| 日志 | 协商和交付日志存在本地 |

### 7.4 离线与异步

- 交付方可离线创建交付意图，联网后自动完成协商和投递
- 接收方离线时，交付包缓存在云端交付总线，上线后推送
- 协商信息（Inbox能力声明）缓存本地，定期更新

## 九、关键指标

| 指标 | 定义 | 目标值 |
|------|------|--------|
| 交付成功率 | 发起交付后最终送达的比例 | > 99% |
| 用户操作次数 | 完成一次交付需要用户点击的次数 | 静默模式下 0 次 |
| 交付时间 | 从产出完成到对方收到的时间 | 实时交付 < 5s |
| 格式转换准确率 | 转换预览后用户选择转换版本的比例 | > 80% |
| 规则匹配准确率 | 文件名关键词匹配正确的比例 | > 95% |
| AI 分类准确率 | 文件内容分类正确的比例 | > 85% |

## 十、开放问题

1. **身份统一**：用户如何跨组织边界使用？用手机号？邮箱？还是需要一个开放的身份协议？（类似 Email 的 `user@domain`）
2. **交付对象的可移植性**：换公司/换学校后，交付对象列表能否迁移？
3. **垃圾交付问题**：如果人人都能无成本交付，如何防止"交付泛滥"？接收方需要什么样的过滤/限流机制？
4. **冷启动**：新用户没有任何交付对象和规则时，体验如何不打断？
5. **商业模型**：谁为这个基础设施付费？个人用户免费 + 企业版收费？还是按交付量计费？
