# Relay（递达）—— 一种数字协作新范式

## 核心洞察

所有可数字化的任务，最终都需要交付给另一个人或系统来闭环。任务本身的数字化（怎么做）已经被充分关注，但**交付层**（做完后怎么到对的收件方手里）仍然停留在邮件/消息模型——人手动指定收件人、人手动点发送。

真正的问题：**每个人不应该关心"发给谁、怎么发、什么格式、什么时候发"——这些应该是基础设施的事。**

## 交付层创新：四个维度

### 一、交付协议创新

#### 1. 静默交付（无感送达）
任务完成后无需主动"发送"，系统自动路由到正确的人/系统。完成即交付。

#### 2. 协商协议（Sender-Inbox 语义握手）
发送前双方 Agent 自动协商格式、通道、时机——TCP 式的语义层握手成为标准交付协议。

#### 3. 可撤回交付（时效性控制）
Agent 直连通道下，接收方未读前发送方可撤回交付物。

### 二、交付对象创新

#### 4. 交付对象（人 + 系统）
交付对象不限于人——可以是 Webhook、IM Bot、或任何系统端点。人的交付对象标记关系类型（上级/老师/学生/客户）。

#### 5. 多态交付（同一产出，不同形态）
一份数据报告：技术看原始数据，老板看可视化图表，审计看合规摘要——同一来源，按收件方自动变换呈现形态。

#### 6. 影子收件人（AI 预处理）
每条交付到达时 AI 自动生成摘要、提取待办项。接收方不用逐条打开就能判断优先级。

### 三、交付物形态创新

#### 7. 活的交付物（持续更新的指针）
类似 Notion 链接 vs Word 附件。收件方看到的永远是"此刻的最新版本"，而不是"发出去那一刻的快照"。

#### 8. 批量交付（同类型聚合）
同一周期、同一收件方、同类型的交付自动打包为一批，接收方看到一条汇总而非多条通知。

### 四、交付关系创新

#### 9. 反向交付（收件方定义要什么）
收件方预先声明"我需要什么样的输入"，生产者做完后系统自动匹配、自动路由。接口定义先行。

#### 10. 跨组织交付
支持 `姓名@组织` 身份格式，跨组织边界的交付自动完成安全校验。

## 核心模型：Relay

### 名字的由来

Relay，接力赛中的交接棒。你只管跑你的一段（做任务），棒交出去的过程自动发生。中文名「递达」——递出去，达得到。

### 模型概览

```
[你做你的任务] → [Sender] →→→ [Inbox] → [对方看到他要的东西]
                                    ↑
                               [对方的 Sender] ← [对方做他的任务]
```

每个人只和两个东西打交道：
- **Sender（递送端）**：你做完的事，它负责送出去
- **Inbox（收件端）**：别人送来的事，它负责整理好给你

你不再需要知道"发谁、发什么格式、什么时候发、发到哪里"——这些都是 Relay 两端之间协商的事。

### Sender（递送端）的职责

| 环节 | 做的事 |
|------|--------|
| 识别 | 这是什么类型的产出？（周报、代码、申请、审批、作业...） |
| 匹配 | 这类东西应该给谁？（从交付对象里推导） |
| 转换 | 对方需要什么格式？（对方的 Inbox 声明了偏好） |
| 时机 | 是现在发、等对方在线发、还是攒一批一起发？ |
| 通道 | 走哪个渠道投递？（对方的 Inbox 注册了可用的入口） |
| 确认 | 对方收到了吗？没收到就重试/换通道 |

### Inbox（收件端）的职责

| 环节 | 做的事 |
|------|--------|
| 接收 | 开放多个入口（邮件入站、Agent 直连、IM、Webhook），统一收口 |
| 分类 | 待处理 / 已归档，按交付类型区分 |
| 转换 | 把对方的格式转成你习惯的格式 |
| 聚合 | 把同类的东西归到一起（所有学生的作业，不是散着的） |
| 预处理 | AI 先帮你读一遍，生成摘要、提取待办 |
| 呈现 | 按你的工作节奏推给你（每日摘要 + 免打扰模式） |

### 关键机制：Sender 和 Inbox 之间的协商协议

TCP 式的语义层握手——双方在"这是什么、怎么传、什么时机传"上达成一致：

1. Sender 查询对方的 Inbox："我要发 X 类型的东西，你怎么收？"
2. Inbox 返回能力声明："我收 Markdown，通道 ID 是 xxx，当前负载状态低"
3. 双方协商完成后，传输发生

### 创新点与模型的关系

| 创新点 | 在模型中的位置 |
|--------|--------------|
| 静默交付 | Sender——完成即触发，不需要人操作 |
| 协商协议 | Sender ↔ Inbox 之间自动握手 |
| 交付对象 | Sender 查询交付对象列表确定收件方 |
| 多态交付 | Sender 按 Inbox 声明偏好做格式转换 |
| AI 预处理 | Inbox 内置 AI 预处理模块 |
| 活的交付物 | Sender 传"指针"，Inbox 持续订阅 |
| 批量交付 | Sender 同类聚合后投递 |
| 反向交付 | Inbox 先声明需求，Sender 按规约提交 |
| 跨组织交付 | Sender/Inbox 支持跨组织寻址和安全校验 |

### 最终体验

> 我只管做。做完交给 Sender。要看什么，打开 Inbox。

所有"该发给谁、怎么发、发了没有、对方收到没、格式对不对、是不是发晚了"——这些认知负担全部消失。Relay 成为每个人的数字代理，把协作从手工操作降为基础设施。

**本质上，是把人从路由层解放出来。**

## 产品形态分析

### 形态 A：独立 App ❌

一个独立的"交付中枢"应用。**致命问题**：产出分散在各处（VS Code、飞书、WPS、邮箱...），独立 App 意味着要手动把东西搬进去，反而增加了负担，违背"只管做，只管发"的原则。

### 形态 B：各 App 内插件 ⚠️

每个应用内嵌一个交付按钮或面板。**可用但碎片化**：每个 App 都要开发适配，推广成本高；插件之间不互通，规则割裂；收件端也要逐 App 检查。

### 形态 C：Agent 产品 ✅（推荐）

一个运行在后台的**智能代理层**，不改变用户现有工作习惯：

```
[本地 Agent 进程]  ←→  [云端协调层]  ←→  [其他人的 Agent]
       ↓                        ↓
 [监听你的产出目录]    [交付对象注册表]
                      [规则引擎：什么该交给谁]
                      [每个人的 Inbox]
```

**为何 Agent 形态最合适**：

| 优势 | 说明 |
|------|------|
| 不改变工作习惯 | 你还在原来的 App 里做事 |
| 后台静默交付 | Agent 监听、识别、转换、投递，你无感 |
| 聚合接收 | 不用逐 App 检查"有没有新东西" |
| 跨 App 能力 | 不管产出在哪个工具，Agent 都能感知 |
| 渐进落地 | 从简单文件监听起步，逐步加 AI 能力 |

## 落地路径

### 第一阶段：MVP（v0.0.1 ~ v0.0.2）

- 拖拽文件到投放卡片 → 交付窗口 → 邮件投递
- 手动建立交付对象列表（人 + 系统）
- 文件名关键词匹配规则模板预填
- 文件夹监听 + 静默交付
- 托盘状态指示 + 交付记录

### 第二阶段：AI 识别 + Inbox 闭环（v0.0.3 ~ v0.1.0）

- AI 文件内容分类 + 反馈学习
- 元数据自动提取 + 收件人智能推荐
- 本地收件箱（待处理/已归档两区）
- 邮件入站监听（IMAP），非 Relay 用户的邮件自动进入收件箱
- 每日摘要通知 + 免打扰模式
- AI 收件预处理（摘要 + 待办提取）

### 第三阶段：格式转换 + 多通道（v0.1.1 ~ v0.2.1）

- 格式偏好声明 + 转换引擎 + 预览
- 通道抽象层：邮件 / IM Bot / Webhook / Agent 直连
- 通道选择策略 + 健康检查
- 企业通讯录导入

### 第四阶段：协商协议（v0.3.0）

- Inbox 能力声明
- 协商握手协议 + 失败处理
- 批处理协商 + 协商日志

### 第五阶段：智能交付 + 反向交付（v0.4.0 ~ v0.4.1）

- 活的交付物（传指针不传副本）+ 可撤回
- 反向交付（需求声明）+ 需求-交付匹配

### 第六阶段：开放生态（v0.5.0）

- 开放交付协议 + Agent SDK
- 跨组织交付

## 本质总结

这不是一个"产品"（单一 App），而是一个**层**——位于日常使用的各种工具之上、底层通信基础设施（邮件/IM/API）之上的**智能路由和转换层**。

每个人的数字秘书（Agent），负责安排"收发"，让每个人只关心"做"。

交互界面可以非常薄——一个系统托盘图标、一个通知摘要、一个分享菜单里的选项。重的逻辑全在后台规则引擎和 AI 识别上。
